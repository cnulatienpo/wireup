Nothing here is an object

"This guide explains TouchDesigner signal flow explained for beginners in plain language. It focuses on practical decisions you can make while building and debugging TouchDesigner networks.",

The habit you bring in with you is object-thinking.

Files, layers, images, clips, shapes. Things that sit there until you pick them up. Things with edges, names, and a sense of completion. Most software trains you to believe that work is about arranging objects into a satisfying arrangement and then stepping back.

That habit causes immediate confusion here.

What looks like an object is not a thing you possess. It is a container that keeps asking a question. Every box in the network is standing there saying, “If something arrives like this, I will respond like that.” Nothing asserts identity. Everything waits for context.

An image node is not an image. It is a place where images pass through. A number is not a value you set once. It is a condition that keeps getting reasserted. A shape is not geometry in the way a file is geometry. It is a description that stays valid as long as nothing upstream contradicts it.

This distinction matters because objects imply ownership. Containers imply obligation. Once you wire something in, the container must deal with it every frame. It does not remember how it felt when you created it. It only knows what it is receiving now.

That is why copying a node does not copy behavior the way people expect. You duplicated a question, not an answer. Without the same surroundings, the question produces a different outcome. Context is not decoration here. It is substance.

Object-based thinking encourages accumulation. Add another layer. Duplicate another clip. Stack until it looks right. Container-based thinking punishes that instinct. Every addition introduces another obligation the system must honor, repeatedly, without complaint.

People often ask where something lives. Where the image is stored. Where the motion exists. Those questions assume a thing sitting somewhere, waiting to be used. What exists instead is a path. Motion exists because something upstream keeps changing. Images exist because something keeps supplying pixels.

This is also why deleting a node can feel disproportionate. You did not remove a thing; you removed a promise. Everything downstream reorganizes itself around that absence immediately. The system does not mourn. It reroutes.

Once you stop looking for objects, frustration drops. You stop hunting for the place where the result lives. You start watching how conditions propagate. The work shifts from collecting things to arranging responsibilities.

Nothing here is an object because objects would slow the system down. What exists instead are agreements that stay active for as long as you keep them wired.
